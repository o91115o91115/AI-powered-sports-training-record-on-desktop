# 第一階段議題處理

## 文件目的

本文件記錄第一階段功能完成後的專案盤點結果，以及進入「多項運動項目」階段前應優先處理的議題。

本次盤點涵蓋：

- 專案結構與既有規劃文件
- 使用者目標設定、AI 計畫、版本調整、訓練月曆與歷史紀錄流程
- Prisma 與 SQLite 資料庫狀態
- AI 輸入輸出格式及風險提醒
- TypeScript、ESLint、Vitest 與正式建置
- 本機實際頁面操作流程

## 盤點結論

目前主要功能已可執行，型別檢查、程式碼檢查、單元測試與正式建置均可完成，但仍有數項會影響資料即時性、統計正確性、版本切換及後續資料庫演進的重大議題。

在完成下列穩定化修正前，不建議直接導入多項運動項目。

## 第一階段重大議題

### 1. 資料頁被正式建置成靜態頁面

#### 現況

`next build` 顯示以下資料頁被預先產生成靜態內容：

- `/`
- `/dashboard`
- `/goals`
- `/planner`
- `/calendar`
- `/history`
- `/logs`

這些頁面會直接讀取本機 SQLite 資料，但目前沒有明確宣告動態渲染。

#### 可能影響

- 頁面可能持續顯示建置當下的舊資料。
- 首次建立使用者與目標後，首頁仍可能保留原本的轉址或空白狀態。
- 訓練、飲食、AI 回饋或計畫版本更新後，不同頁面的資料可能不同步。
- 現有 `revalidatePath` 涵蓋範圍不完整。例如目標儲存只重新驗證 `/goals`，部分紀錄操作只重新驗證 `/dashboard`，但實際總覽路徑也包含 `/`。

#### 建議處理

1. 對直接讀取 Prisma 的資料頁統一採用適合本機資料庫的動態渲染策略。
2. 盤點每個 Server Action 實際影響的頁面，補齊 `/`、`/planner`、`/calendar`、`/history` 等重新驗證路徑。
3. 增加首次設定完成後的首頁導向測試，以及資料更新後重新整理的驗證。

### 2. Prisma migration 歷史與實際資料庫不一致

#### 現況

目前 SQLite 資料庫已具有完整資料表與後續新增欄位，但 `prisma migrate status` 顯示下列三個 migration 均尚未套用：

- `202607011_create_initial_tables`
- `20260711210000_add_training_plan_conversations`
- `20260718140000_add_adjustment_conversations`

`scripts/init-sqlite-db.py` 會直接執行 migration SQL，但不會建立或更新 Prisma 的 `_prisma_migrations` 紀錄。

#### 可能影響

- 後續新增運動類型與欄位時可能發生 schema drift。
- `prisma migrate dev` 可能偵測到資料庫與 migration 歷史不一致。
- Prisma 可能要求重設資料庫，造成本機既有訓練與飲食資料風險。
- 不同開發環境可能建立出不同結構的資料庫。

#### 建議處理

1. 操作前先備份 `prisma/dev.db`。
2. 比對現有資料庫結構、`schema.prisma` 與三份 migration SQL。
3. 確認結構一致後，建立安全的 migration baseline，不直接重設目前資料庫。
4. 修正初始化方式，使新環境一律透過 Prisma migration 建立資料庫。
5. 更新 README 與本機啟動文件，避免同時存在兩套資料庫初始化流程。

#### 相關檔案

- `prisma/schema.prisma`
- `prisma/migrations/`
- `scripts/init-sqlite-db.py`
- `scripts/inspect-db-schema.py`
- `README.md`

### 3. 首頁完成率可能超過 100%

#### 現況

實際頁面走查時，首頁顯示：

```text
本週完成率：150%
已回報 3 / 2 個訓練日
```

目前公式的分母會排除休息日，但分子會計入本週所有實際訓練紀錄日期，包括原本安排為休息日的日期，而且沒有完整限制於目前計畫的有效非休息訓練日。

#### 可能影響

- 完成率超過 100%，使用者無法正確判斷計畫執行情況。
- 在休息日額外訓練時，系統可能把偏離計畫誤算成較高完成度。
- 多運動導入後，同一天多次訓練會讓完成率定義更加混亂。

#### 建議處理

1. 先定義完成率是依「計畫訓練日」、「訓練課次」或「運動項目」計算。
2. 現階段至少將分子限制為目前版本、本週、非休息訓練日且已有有效完成狀態的日期。
3. 額外訓練應獨立顯示，不直接增加原計畫完成率。
4. 完成率上限應為 100%，並增加休息日額外訓練與同日多筆紀錄測試。

#### 相關檔案

- `src/app/dashboard/dashboard-content.tsx`
- `src/components/dashboard/dashboard-charts.tsx`

### 4. 切換計畫版本後，AI 回饋與月曆紀錄來源不一致

#### 現況

月曆會依「使用者與日期」聚合舊版本建立的訓練及飲食紀錄，讓使用者切換計畫版本後仍能看到原有紀錄。

但產生 AI 今日回饋時，後端只讀取目前版本 `TrainingDay` 直接關聯的 `WorkoutLog` 與 `FoodLog`。

實際資料確認結果：目前 V4 的 `2026-07-17` 訓練日直接關聯訓練紀錄與飲食紀錄均為 0 筆，但月曆仍會顯示建立於 V2 的同日紀錄。

#### 可能影響

- 月曆明明顯示已有紀錄，重新產生 AI 回饋時卻可能回覆「目前沒有實際紀錄」。
- AI 可能忽略舊版本留下的同日訓練、飲食與疼痛資料。
- 計畫調整越頻繁，月曆顯示與 AI 分析依據越容易不一致。
- 多運動階段若同一天有多種運動，問題會進一步放大。

#### 建議處理

1. 統一月曆與 AI 回饋的紀錄查詢規則。
2. 短期可讓 AI 回饋依 `userProfileId + logDate` 聚合同日紀錄，再驗證日期屬於目前計畫。
3. 中期應評估將「實際運動紀錄」與特定計畫版本解耦，計畫版本只保留規劃內容。
4. 增加版本切換後重新產生 AI 回饋的整合測試。

#### 相關檔案

- `src/app/calendar/page.tsx`
- `src/app/calendar/review-actions.ts`
- `src/components/training/ai-feedback-panel.tsx`
- `src/components/training/workout-log-manager.tsx`
- `src/components/training/food-log-manager.tsx`

### 5. 多運動的「目前計畫」與並行規則尚未定義

#### 現況

目前資料模型、表單、AI schema 與 Prompt 主要以跑步及馬拉松為核心：

- 目標包含週跑量、5K、10K、半馬及全馬經驗。
- 訓練類型包含輕鬆跑、長跑、節奏跑、間歇、休息、交叉訓練與比賽。
- 月曆及 Dashboard 只會選取一個最近更新的 `active` 計畫。
- 確認一個新計畫版本時，不會自動停用其他計畫。

目前實際訓練紀錄已可用文字記錄游泳等活動，但正式計畫、目標、統計與 AI 建議仍以跑步邏輯處理，這不等同於完整支援多運動。

#### 導入多運動前需決定

1. 同時間只能有一個跨運動計畫。
2. 每個運動項目可各有一個進行中計畫。
3. 使用一個複合計畫，同一天可包含多個運動課次。

上述選擇會影響：

- `TrainingGoal`、`TrainingPlan`、`TrainingDay` 與 `WorkoutLog` 的資料欄位
- 月曆一天是否能包含多個課次
- Dashboard 完成率與運動量統計
- AI Prompt 所需的運動專屬能力資料
- 疼痛、疲勞與跨運動負荷判斷
- 計畫版本與實際紀錄的關聯方式

#### 建議方向

為保留未來擴充性，建議採用「一個複合計畫可包含多項運動課次」的方向，並使每筆計畫課次及實際紀錄都有明確 `sportType`。運動專屬數值應按類型設計，不應直接把所有跑步欄位套用到游泳、自行車或重量訓練。

## 其他觀察事項

以下項目不是目前最優先阻塞，但建議在第一階段穩定化時一併確認：

1. 歷史紀錄在沒有時間資料時會顯示「未提供 分鐘」，可調整為較自然的提示。
2. `TrainingPlanConversation.trainingPlanId` 沒有資料庫外鍵，需由應用程式維持一致性。
3. 系統允許多個計畫同時為 `active`，但月曆只選擇最近更新的一個。
4. 核心 Server Action、版本切換與完整使用者流程目前缺少自動化整合測試。
5. AI 回饋可重複產生多筆，後續可評估是否需要顯示版本或避免短時間內重複生成。

## 建議處理順序

### 第一批：資料與頁面穩定性

1. 修正資料頁的動態渲染策略。
2. 補齊各 Server Action 的頁面重新驗證範圍。
3. 修正首頁完成率公式。

### 第二批：資料庫安全

1. 備份現有 SQLite 資料庫。
2. 比對現有 schema 與 migration SQL。
3. 建立 Prisma migration baseline。
4. 統一資料庫初始化與升級方式。

### 第三批：版本與紀錄一致性

1. 統一月曆、Dashboard、歷史紀錄與 AI 回饋的紀錄查詢規則。
2. 修正計畫版本切換後的同日紀錄聚合。
3. 補充版本切換與 AI 回饋整合測試。

### 第四批：多運動設計確認

1. 決定單一計畫或多計畫並行模式。
2. 決定第一批支援的運動項目。
3. 定義共同欄位與運動專屬欄位。
4. 定義同日多課次、完成率與跨運動負荷計算方式。
5. 完成資料相容與 migration 設計後再開始實作。

## 進入下一階段的完成條件

符合以下條件後，再開始導入多項運動項目：

- 資料頁更新後能穩定顯示最新 SQLite 資料。
- 首次設定、訓練紀錄、飲食紀錄、版本確認及 AI 回饋流程均可正常執行。
- 完成率不會超過 100%，額外訓練有獨立且明確的呈現方式。
- 計畫版本切換後，月曆與 AI 回饋使用相同的實際紀錄來源。
- Prisma migration 歷史與現有資料庫一致，且可安全新增 migration。
- 多運動計畫模式、第一批運動項目與運動專屬欄位已確認。
