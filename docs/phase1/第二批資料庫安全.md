# 第一階段第二批：資料庫安全

## 文件目的

本文件記錄 SQLite 備份、Prisma migration baseline、乾淨部署驗證及 Server 備份路徑的處理結果。

## 完成內容

### 1. 可攜式備份設定

備份路徑不綁定特定作業系統，透過環境變數控制：

```env
DATABASE_BACKUP_DIR="./data/backups"
DATABASE_BACKUP_RETENTION="7"
```

- 相對路徑以專案根目錄為基準。
- Windows、Linux 與 Docker 可改用絕對路徑。
- `NODE_ENV=production` 時若未設定 `DATABASE_BACKUP_DIR`，備份工具會拒絕執行。
- 備份資料不納入 Git。

Server 範例：

```env
# Windows Server
DATABASE_BACKUP_DIR="D:\AITrainingPlanner\backups"

# Linux Server
DATABASE_BACKUP_DIR="/var/lib/ai-training-planner/backups"

# Docker persistent volume
DATABASE_BACKUP_DIR="/app-data/backups"
```

### 2. 一致性備份

`pnpm db:backup` 使用 SQLite `VACUUM INTO` 建立交易一致的快照，並產生 manifest：

- 備份時間
- 來源與目的路徑
- SHA-256
- 檔案大小
- `integrity_check`
- 外鍵違規數量
- 各資料表筆數
- migration 歷史

備份依時間命名且不覆寫，預設保留最近 7 份。

### 3. Migration baseline

處理前狀態：

- 實際資料表已存在。
- `schema.prisma` 與正式資料庫結構差異為空。
- `_prisma_migrations` 不存在。
- Prisma 將四份 migration 判定為未套用。

完成一致性備份後，依順序將四份既有 migration 標記為已套用。既有 SQL 沒有重新執行，也沒有重設資料庫。

乾淨部署測試另外發現，早期 migration 的 `updatedAt` 資料庫預設值與 Prisma `@updatedAt` 有差異，因此新增：

```text
20260719170000_align_updated_at_defaults
```

正式資料庫原本已是目標結構，所以只將這份 migration 標記為已套用；新環境會實際執行它，使 migration 建立結果與 Schema 一致。

完成後：

- `_prisma_migrations` 已建立。
- migration 狀態為 5/5。
- 正式資料庫 schema diff 為空。
- 空白暫存資料庫可從零套用五份 migration，schema diff 亦為空。

### 4. 統一初始化流程

新環境或 Server 部署：

```powershell
pnpm install
pnpm db:deploy
pnpm db:verify
```

不再使用 Python 直接執行 migration SQL。`db:deploy` 會在需要時建立空白 SQLite 檔案，再交由 Prisma 套用所有 migration。

### 5. 驗證內容

`pnpm db:verify` 會檢查：

- SQLite 完整性
- 外鍵違規
- 必要資料表
- migration 完成狀態
- migration SQL checksum
- `sportCategory` 標準代碼
- 無效 `activeVersionId`
- 對話與計畫的孤兒關聯
- 尚待確認分類的資料數量

`pnpm db:verify-clean` 會在系統暫存目錄建立空白資料庫，套用全部 migration、確認 schema diff，最後只刪除該暫存資料庫。

## 復原流程

資料庫損壞或 migration 失敗時，不要執行 reset：

1. 停止 Next.js Server，避免新資料寫入。
2. 使用 `node scripts/verify-sqlite-db.cjs --database <備份檔>` 驗證備份。
3. 使用作業系統的 SHA-256 工具比對 manifest。
4. 將目前的 `prisma/dev.db` 改名保留，不直接覆寫或刪除。
5. 將已驗證的備份複製為 `prisma/dev.db`。
6. 執行 `pnpm db:verify` 與 `pnpm db:status`。
7. 確認頁面與主要資料後再恢復服務。

實際復原會替換正式資料庫，必須由使用者明確確認後才能執行。

## 禁止事項

- 不可對含有使用者資料的資料庫執行 `prisma migrate reset`。
- 不可使用 `prisma db push` 取代 migration。
- 不可修改已套用 migration 的 SQL。
- 不可把 SQLite、備份、manifest 或真實環境變數提交到 Git。
- 不可把 Docker 容器內未掛載的目錄當作持久化備份位置。
