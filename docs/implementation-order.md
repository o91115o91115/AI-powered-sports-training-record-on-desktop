# 模組實作執行順序建議

## 1. 文件目的

本文件依據 `project-overview.md`、`technical-stack.md`、`database-design.md` 與 `local-server-startup.md`，整理 AI 運動訓練與飲食紀錄系統的建議實作順序。

實作原則：

1. 先完成資料結構與基礎流程，再開發 AI 功能。
2. 先建立可儲存、可查詢、可確認的核心資料，再做分析與重新調整。
3. AI 產生內容必須先經 schema 驗證與使用者確認，不直接覆蓋正式資料。
4. 訓練計畫調整必須保留版本，不得直接覆蓋原始計畫。
5. 每個階段都應能獨立驗收，避免一次開發過多模組造成錯誤難以定位。

---

## 2. 建議總順序

1. 基礎架構與資料庫
2. 使用者資料與目標設定模組
3. 訓練計畫與版本管理模組
4. AI 訓練菜單規劃模組
5. 每日營養規劃模組
6. 訓練月曆與今日任務顯示
7. 每日運動紀錄模組
8. 每日飲食紀錄模組
9. Logging Agent 紀錄整理
10. AI 分析與回饋模組
11. 訓練菜單重新調整模組
12. Dashboard 與歷史紀錄整合

---

## 3. 階段一：基礎架構與資料庫

### 目標

確認專案可正常啟動，並建立後續模組共用的資料庫、型別、環境變數與基礎 API 結構。

### 主要工作

1. 確認 Next.js、React、TypeScript 專案可啟動。
2. 確認 `.env.local` 設定方式。
3. 建立 Prisma schema 與 SQLite 連線。
4. 建立 `UserProfile`、`TrainingGoal`、`TrainingPlan` 等第一版資料模型。
5. 建立 Prisma client 共用入口。
6. 確認 health API 可正常回應。

### 前置依賴

- 無，這是所有模組的基礎。

### 完成標準

1. `pnpm dev` 可啟動本機服務。
2. health API 可正常回傳。
3. Prisma schema 可通過 validate。
4. 本機資料庫可建立或連線。

### 注意事項

- `local-server-startup.md` 提到 Prisma migration / db push 曾出現 schema engine 空白錯誤，應在開始資料庫讀寫功能前優先排除。
- 不要在此階段加入非必要外部服務。

---

## 4. 階段二：使用者資料與目標設定模組

### 目標

建立 AI 規劃訓練與營養建議所需的基礎資料。

### 主要工作

1. 建立使用者基本資料表單。
2. 建立訓練目標表單。
3. 儲存目前跑步能力、可訓練時間、傷痛、疲勞與飲食限制。
4. 建立基本資料與目標的讀取、更新 API。
5. 加入必要欄位驗證與錯誤提示。

### 對應資料表

- `UserProfile`
- `TrainingGoal`

### 前置依賴

- 基礎架構與資料庫。

### 完成標準

1. 使用者可以建立基本資料。
2. 使用者可以建立訓練目標。
3. 儲存後重新整理頁面仍可讀取資料。
4. 必填欄位缺漏時有明確提示。

### 注意事項

- 傷痛、疲勞與飲食限制會影響 AI 建議，資料欄位不可任意省略。
- 不要在此階段直接產生 AI 計畫。

---

## 5. 階段三：訓練計畫與版本管理模組

### 目標

建立訓練計畫主檔、版本、每日訓練內容與每日營養建議的儲存結構。

### 主要工作

1. 建立訓練計畫主檔 CRUD。
2. 建立訓練計畫版本 CRUD。
3. 建立每日訓練內容資料讀寫。
4. 建立每日營養建議資料讀寫。
5. 建立草稿、確認、封存等狀態流程。
6. 確認正式計畫只能由使用者確認後套用。

### 對應資料表

- `TrainingPlan`
- `TrainingPlanVersion`
- `TrainingDay`
- `NutritionSuggestion`

### 前置依賴

- 使用者資料與目標設定模組。

### 完成標準

1. 可以建立訓練計畫草稿。
2. 可以建立計畫版本。
3. 可以儲存每日訓練內容與營養建議。
4. 使用者確認後，計畫狀態可更新為正式套用。

### 注意事項

- 訓練計畫不可直接覆蓋，後續調整必須建立新版本。
- `activeVersionId` 的更新應明確，只在使用者確認套用後執行。

---

## 6. 階段四：AI 訓練菜單規劃模組

### 目標

根據使用者資料與訓練目標，透過 Planning Agent 產生訓練與營養計畫草稿。

### 主要工作

1. 建立 OpenAI client 後端封裝。
2. 建立 Planning Agent prompt。
3. 定義 AI 訓練計畫輸出 JSON schema。
4. 使用 zod 驗證 AI 回傳格式。
5. 將 AI 結果儲存為 `TrainingPlanVersion` 草稿。
6. 顯示草稿內容供使用者確認。

### 對應 AI Agent

- Planning Agent

### 前置依賴

- 使用者資料與目標設定模組。
- 訓練計畫與版本管理模組。

### 完成標準

1. AI 可根據使用者目標產生訓練計畫草稿。
2. AI 輸出格式通過 schema 驗證。
3. 訓練計畫草稿不會直接成為正式計畫。
4. 傷痛、疲勞與訓練量安全限制有被納入 prompt。

### 注意事項

- AI 不得宣稱保證完賽或保證成績。
- 若資料不足，AI 應提示需要更多紀錄或補充資料。
- 使用者確認前不得正式套用計畫。

---

## 7. 階段五：每日營養規劃模組

### 目標

依照每日訓練內容產生飲食與補給方向，並與訓練日綁定。

### 主要工作

1. 建立每日營養建議顯示區塊。
2. 將碳水、蛋白質、水分、訓練前後飲食與長跑補給分區顯示。
3. 將營養建議與 `TrainingDay` 對應。
4. 在畫面上標示營養數值為估算或方向性建議。

### 對應資料表

- `NutritionSuggestion`
- `TrainingDay`

### 前置依賴

- 訓練計畫與版本管理模組。
- AI 訓練菜單規劃模組。

### 完成標準

1. 每個訓練日可以顯示對應營養建議。
2. 休息日、長跑日、高強度日的營養建議可區分。
3. 畫面清楚標示非醫療或營養師診斷。

### 注意事項

- 初期不追求精準食物資料庫。
- 不提供極端節食或過度限制熱量建議。

---

## 8. 階段六：訓練月曆與今日任務顯示

### 目標

讓使用者能清楚查看今日訓練、今日營養建議與完整訓練月曆。

### 主要工作

1. 建立今日任務顯示。
2. 建立訓練月曆頁。
3. 標示訓練類型、休息日、長跑日、高強度日。
4. 顯示完成狀態。
5. 支援計畫版本切換或查看。

### 前置依賴

- 訓練計畫與版本管理模組。
- 每日營養規劃模組。

### 完成標準

1. 使用者可以看到今日訓練內容。
2. 使用者可以看到今日營養建議。
3. 月曆可顯示完整訓練安排。
4. 不同訓練狀態有清楚標示。

### 注意事項

- 此階段重點是查看計畫，不處理紀錄輸入。
- 重要風險提醒不可隱藏在不明顯位置。

---

## 9. 階段七：每日運動紀錄模組

### 目標

讓使用者記錄每日實際運動內容，並與原定訓練比對。

### 主要工作

1. 建立運動紀錄輸入表單。
2. 儲存使用者原始輸入。
3. 儲存運動類型、距離、時間、配速、心率。
4. 儲存疲勞分數、疼痛部位、疼痛分數。
5. 儲存完成狀態。
6. 與當日 `TrainingDay` 關聯。

### 對應資料表

- `WorkoutLog`
- `TrainingDay`

### 前置依賴

- 訓練月曆與今日任務顯示。

### 完成標準

1. 使用者可以新增每日運動紀錄。
2. 使用者原始輸入會被保留。
3. 紀錄可以關聯原定訓練日。
4. 疲勞與疼痛欄位可正常儲存。

### 注意事項

- 疼痛與高疲勞資料是後續 AI 風險提醒的重要依據。
- 不要在此階段直接做 AI 診斷。

---

## 10. 階段八：每日飲食紀錄模組

### 目標

讓使用者記錄每日飲食內容，並保留原始輸入與估算資料。

### 主要工作

1. 建立飲食紀錄輸入表單。
2. 儲存使用者原始輸入。
3. 儲存餐別、食物項目、碳水估算、蛋白質估算與熱量估算。
4. 與當日訓練或運動紀錄關聯。
5. 顯示營養估算說明。

### 對應資料表

- `FoodLog`
- `WorkoutLog`
- `TrainingDay`

### 前置依賴

- 每日運動紀錄模組。
- 每日營養規劃模組。

### 完成標準

1. 使用者可以新增飲食紀錄。
2. 原始飲食輸入會被保留。
3. 營養數值清楚標示為估算。
4. 紀錄可與同日訓練脈絡關聯。

### 注意事項

- 若份量不足，不應產生過度精準估算。
- 不要將飲食建議包裝成醫療或營養師診斷。

---

## 11. 階段九：Logging Agent 紀錄整理

### 目標

將使用者自然語言輸入的運動與飲食內容整理成結構化紀錄。

### 主要工作

1. 建立 Logging Agent prompt。
2. 定義運動紀錄與飲食紀錄的 AI 輸出 schema。
3. 將自然語言解析為 `WorkoutLog` 與 `FoodLog` 草稿。
4. 顯示 AI 整理結果供使用者確認。
5. 使用者確認後才寫入正式紀錄。
6. 若資訊不足，提示使用者補充。

### 對應 AI Agent

- Logging Agent

### 前置依賴

- 每日運動紀錄模組。
- 每日飲食紀錄模組。

### 完成標準

1. 使用者可以輸入一段自然語言。
2. AI 可以整理運動與飲食資訊。
3. 缺漏資訊會被標示。
4. 使用者確認前不寫入正式紀錄。

### 注意事項

- 不確定的數值不可當成精準結果。
- AI 不可自行新增使用者沒有提到的訓練或食物。

---

## 12. 階段十：AI 分析與回饋模組

### 目標

比對原定計畫、實際運動紀錄與飲食紀錄，產生今日或本週回饋。

### 主要工作

1. 建立 Review Agent prompt。
2. 取得原定訓練、實際運動紀錄與飲食紀錄。
3. 分析完成度、跑量、強度、疲勞、疼痛與營養落差。
4. 產生 AI 回饋。
5. 儲存回饋內容。
6. 判斷是否建議重新調整計畫。

### 對應資料表

- `AiFeedback`
- `WorkoutLog`
- `FoodLog`
- `TrainingDay`

### 對應 AI Agent

- Review Agent

### 前置依賴

- 每日運動紀錄模組。
- 每日飲食紀錄模組。
- Logging Agent 紀錄整理。

### 完成標準

1. 完成每日紀錄後可以產生 AI 回饋。
2. 回饋包含訓練分析、營養分析、風險提醒與下一步建議。
3. 疲勞或疼痛高風險時會給出保守建議。
4. 回饋可被儲存並查詢。

### 注意事項

- AI 不得做醫療診斷。
- 高風險狀況應建議降低強度或尋求專業協助。

---

## 13. 階段十一：訓練菜單重新調整模組

### 目標

根據近期紀錄、疲勞、傷痛、缺課或目標變更，建立新的訓練計畫版本。

### 主要工作

1. 建立 Replanning Agent prompt。
2. 整理目前計畫與近期紀錄。
3. 產生調整原因、受影響日期與調整後方案。
4. 顯示調整前後差異。
5. 建立 `PlanAdjustment` 草稿。
6. 建立新的 `TrainingPlanVersion` 草稿。
7. 使用者確認後才套用新版本。

### 對應資料表

- `PlanAdjustment`
- `TrainingPlanVersion`
- `TrainingDay`
- `NutritionSuggestion`

### 對應 AI Agent

- Replanning Agent

### 前置依賴

- 訓練計畫與版本管理模組。
- AI 分析與回饋模組。

### 完成標準

1. 系統可以根據近期狀況產生調整建議。
2. 使用者可以查看調整原因與前後差異。
3. 原始計畫不會被覆蓋。
4. 使用者確認後才更新 active version。

### 注意事項

- 不可為了補進度而短期大幅增加跑量。
- 傷痛或疲勞高時，應優先降低訓練負荷。
- 需要保留 `aiModel` 與 `promptSnapshot` 以利追蹤。

---

## 14. 階段十二：Dashboard 與歷史紀錄整合

### 目標

整合各模組資料，讓使用者快速查看今日任務、近期趨勢、AI 建議與歷史紀錄。

### 主要工作

1. 建立 Dashboard 首頁。
2. 顯示今日訓練與營養建議。
3. 顯示本週完成率與跑量。
4. 顯示疲勞、傷痛與 AI 建議摘要。
5. 建立歷史紀錄查詢頁。
6. 查詢過去運動紀錄、飲食紀錄、AI 回饋與計畫調整紀錄。

### 前置依賴

- 訓練月曆與今日任務顯示。
- 每日運動紀錄模組。
- 每日飲食紀錄模組。
- AI 分析與回饋模組。
- 訓練菜單重新調整模組。

### 完成標準

1. 使用者可以在首頁掌握今日任務。
2. 使用者可以看到近期訓練狀態。
3. 使用者可以查詢歷史紀錄。
4. 重要風險提醒顯示清楚。

### 注意事項

- Dashboard 是整合頁面，應避免在資料模組尚未穩定前過早複雜化。
- 不要在 Dashboard 直接修改核心資料，除非流程已明確設計。

## 15. 建議里程碑

## 15.1 MVP 里程碑一：可建立目標與計畫草稿

包含階段：

1. 基礎架構與資料庫。
2. 使用者資料與目標設定模組。
3. 訓練計畫與版本管理模組。
4. AI 訓練菜單規劃模組。

驗收重點：

1. 使用者可以建立目標。
2. AI 可以產生訓練計畫草稿。
3. 使用者確認後可以套用計畫。

## 15.2 MVP 里程碑二：可查看與執行每日訓練

包含階段：

1. 每日營養規劃模組。
2. 訓練月曆與今日任務顯示。
3. 每日運動紀錄模組。
4. 每日飲食紀錄模組。

驗收重點：

1. 使用者可以查看每日訓練與營養建議。
2. 使用者可以記錄運動與飲食。
3. 紀錄可以與原定訓練關聯。

## 15.3 MVP 里程碑三：可由 AI 整理與分析紀錄

包含階段：

1. Logging Agent 紀錄整理。
2. AI 分析與回饋模組。

驗收重點：

1. 使用者可以用自然語言輸入紀錄。
2. AI 可以整理成結構化資料。
3. AI 可以產生訓練、營養與風險回饋。

## 15.4 MVP 里程碑四：可重新調整計畫並追蹤歷史

包含階段：

1. 訓練菜單重新調整模組。
2. Dashboard 與歷史紀錄整合。

驗收重點：

1. AI 可以產生調整建議。
2. 調整後建立新版本，不覆蓋舊版本。
3. 使用者可以查詢歷史紀錄與計畫變化。

---

## 16. 初期不建議優先處理項目

以下項目可延後，避免第一版範圍過大：

1. 其他運動項目，例如網球、游泳、自行車。
2. Garmin、Strava、Apple Health 等外部資料串接。
3. 社群功能。
4. 教練端功能。
5. 付費功能。
6. 精準食物資料庫。
7. 完整 AI 對話歷史管理。
8. 獨立風險事件追蹤表。

---

## 17. 開發注意事項

1. 每次修改 AI prompt、AI schema 或資料庫欄位時，需同步確認前端解析與儲存流程。
2. AI 輸出必須固定格式，避免前端難以解析。
3. 使用者自然語言原始輸入必須保留。
4. 飲食與熱量資料應標示為估算。
5. 傷痛、胸悶、頭暈、呼吸異常與高疲勞狀況應觸發保守建議。
6. 訓練計畫調整必須建立新版本。
7. 每個階段完成後應至少驗證主要流程、錯誤提示與資料是否可正確儲存。
