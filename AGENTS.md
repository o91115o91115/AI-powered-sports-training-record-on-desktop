# Codex 開發規則：AI 訓練菜單規劃系統

## 1. 基本原則

本專案為本機端使用的【AI 訓練菜單規劃】系統，功能重點包含訓練紀錄、飲食紀錄、AI 菜單建議、成效追蹤與風險提醒。

Codex 執行任何開發任務前，必須遵守以下原則：

1. 不得在未理解需求的情況下直接修改程式。
2. 不得自行新增未被要求的主要功能。
3. 不得擅自更改既有系統架構、資料結構或操作流程。
4. 所有修改應以「最小必要異動」為原則。
5. 優先維持系統穩定、資料安全、可讀性與後續維護性。
6. 若需求不明確，應先列出理解內容與待確認事項，不得自行猜測後大幅修改。

---

## 2. 功能理解規則

執行開發前，Codex 必須先閱讀並理解以下內容：

1. 使用者需求描述
2. 既有功能規劃文件
3. 相關頁面、元件、資料欄位與流程
4. 目前程式碼的實作方式
5. 本次任務與既有功能的關聯

在開始修改前，Codex 必須先輸出以下內容：

```md
## 任務理解

### 本次目標
- 說明本次要完成的功能或調整目的。

### 影響範圍
- 可能影響的頁面、模組、API、資料欄位或 AI 輸出。

### 不處理範圍
- 本次任務不會處理的項目，避免擴大修改範圍。

### 需確認事項
- 若需求有不清楚或可能影響架構的地方，需明確列出。
```

若任務可直接判斷，仍需簡短說明理解內容後再執行。

---

## 3. 調整事項與異動檔案規則

每次修改前，Codex 必須先規劃預計調整內容。

格式如下：

```md
## 預計調整事項

1. 調整項目：
   - 說明要修改的功能或程式邏輯。

2. 預計異動檔案：
   - `path/to/file`
     - 異動原因：
     - 預計修改內容：

3. 可能影響：
   - 說明是否影響既有流程、資料格式、畫面顯示或 AI 輸出。
```

完成修改後，必須回報實際異動檔案。

格式如下：

```md
## 實際異動檔案

- `path/to/file`
  - 完成內容：
  - 是否與原規劃一致：
  - 若有差異，說明原因：
```

不得只回覆「已完成」而不說明修改內容。

## 4. 程式註解規則

註解必須簡潔、明確，讓後續開發人員能快速理解原因與用途。

### 應該加註解的情況

1. AI Prompt 組裝邏輯
2. 訓練菜單產生規則
3. 飲食建議或熱量計算邏輯
4. 風險提醒或安全限制邏輯
5. 資料轉換、資料清洗或格式化邏輯
6. 非直覺的判斷條件
7. 為避免錯誤而加入的防呆處理

### 註解風格

應使用簡短註解：

```ts
// 根據使用者目標與近期訓練紀錄產生建議菜單
```

避免無意義註解：

```ts
// 設定變數
// 呼叫 function
// 如果是 true 就進入
```

避免過度冗長註解：

```ts
// 這邊會透過很多資料進行非常完整的判斷，然後依照不同條件產生結果...
```

### 註解原則

1. 說明「為什麼這樣做」，不要只描述程式表面行為。
2. 不要為每一行程式加註解。
3. 不要留下過期、錯誤或與程式不一致的註解。
4. 若修改邏輯，必須同步更新相關註解。
5. AI Prompt、風險規則、輸出格式限制應特別保留清楚註解。

---

## 5. AI 功能開發規則

本系統涉及 AI 訓練與飲食建議，Codex 修改相關功能時必須遵守以下規則：

1. AI 輸出不得宣稱為醫療診斷。
2. 使用者若有疼痛、受傷、過度疲勞等情況，系統應提醒降低強度或尋求專業協助。
3. 訓練菜單應避免突然大幅增加跑量、重量或訓練頻率。
4. 飲食建議應避免極端節食、過度限制熱量或不安全建議。
5. AI 回覆應清楚標示建議依據，例如：

   * 使用者目標
   * 近期訓練紀錄
   * 疲勞狀態
   * 飲食紀錄
   * 成效變化
6. AI 輸出格式應固定，避免前端難以解析。
7. 若 AI 資料不足，應回覆「需要更多紀錄」而不是自行編造。

---

## 6. 資料與紀錄處理規則

處理訓練紀錄、飲食紀錄、身體狀態或使用者資料時，必須遵守：

1. 不得刪除既有資料，除非任務明確要求。
2. 不得任意改變資料欄位名稱或格式。
3. 若需新增欄位，必須說明：

   * 欄位用途
   * 欄位型態
   * 是否為必填
   * 對既有資料的影響
4. 若修改資料格式，必須考慮舊資料相容性。
5. 本機端資料應避免暴露敏感資訊。
6. 測試資料不得使用真實個資。

---

## 7. 畫面與操作流程規則

調整畫面時，Codex 必須注意：

1. 操作流程應簡單清楚。
2. 不得讓使用者需要理解技術細節才能操作。
3. 訓練紀錄、飲食紀錄、AI 建議與成效追蹤應有明確區分。
4. 重要風險提醒不可隱藏在不明顯位置。
5. 若新增按鈕、頁籤或區塊，需說明其用途。
6. 不得為了單一需求造成畫面流程混亂。

---

## 8. 錯誤處理規則

Codex 必須確保常見錯誤情境有基本處理：

1. 使用者未輸入必要資料
2. AI 回傳格式錯誤
3. 訓練紀錄不足
4. 飲食紀錄不足
5. 本機資料讀取失敗
6. 本機資料儲存失敗
7. 數值格式錯誤，例如距離、時間、重量、熱量
8. 無法產生合理菜單

錯誤訊息應讓使用者知道下一步該怎麼做。

避免只顯示：

```md
Error
發生錯誤
系統異常
```

應改為：

```md
目前訓練紀錄不足，請先新增至少一筆訓練紀錄後再產生菜單。
```

---

## 9. 測試與確認規則

完成修改後，Codex 必須至少檢查：

1. 本次修改功能是否可正常執行
2. 是否影響既有主要流程
3. 是否有明顯畫面錯誤
4. 是否有格式錯誤或資料無法儲存
5. AI 輸出格式是否符合前端需求
6. 錯誤情境是否有合理提示

回報時需列出測試結果：

```md
## 測試結果

- 測試項目：
- 測試方式：
- 測試結果：
- 尚未測試項目：
- 未測試原因：
```

若無法執行測試，必須明確說明原因，不得假裝已測試。

---

## 11. 禁止事項

Codex 不得執行以下行為：

1. 未說明原因就大幅重構。
2. 未經要求刪除功能。
3. 未經要求改變使用流程。
4. 未經要求改變資料格式。
5. 未經要求新增外部服務依賴。
6. 未經要求加入雲端同步、帳號登入或付費服務。
7. 未經要求修改專案設定檔。
8. 留下無用的測試程式、假資料或 debug log。
9. 產生與實際程式不一致的文件。
10. 宣稱完成但未確認功能狀態。

---

## 12. 優先順序

若規則之間發生衝突，依照以下順序判斷：

1. 使用者明確指令
2. 系統安全與資料保護
3. AI 輸出安全限制
4. 既有功能穩定性
5. 最小必要異動
6. 程式可讀性
7. 開發效率
